## Overview

This section covers the generation of `checksums.txt` inside the GitHub Actions release job. After all three binary artifacts are downloaded, a single `sha256sum` command produces the checksum file. The file is then uploaded alongside the binaries as a GitHub Release asset.

The `checksums.txt` file serves as a corruption/truncation guard when a user runs `install.sh`. It is not a tamper-proof guarantee (see Security Limitation below).

## Implementation

### File to modify

```
.github/workflows/release.yml
```

The checksum generation step belongs in the `release` job, after the `actions/download-artifact@v4` step and before the `softprops/action-gh-release@v2` step.

### Where it fits in the release job

The release job structure (from section-01-ci-workflow) is:

```
release(needs: build):
  - download-artifact@v4  (merge-multiple: true, path: artifacts/)
  - [THIS SECTION] sha256sum step
  - action-gh-release@v2  (files: artifacts/* checksums.txt)
```

### The sha256sum step

Add a `run` step between the artifact download and the release action:

```yaml
- name: Generate checksums
  run: sha256sum artifacts/* > checksums.txt
```

The `artifacts/` directory is populated by `actions/download-artifact@v4` with `merge-multiple: true`, so all three binaries land flat in that directory. Running `sha256sum artifacts/*` produces lines like:

```
e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855  artifacts/jira-assistant-linux-x64
a87ff679a2f3e71d9181a67b7542122c...  artifacts/jira-assistant-macos-arm64
...
```

Note the filename in each line will include the `artifacts/` prefix because `sha256sum` records the path as given. The `install.sh` checksum verification (section-04-install-core) must account for this when grepping for the binary name — using a pattern match on the basename rather than a full path match.

If this prefix is undesirable, an alternative form changes directory first:

```yaml
- name: Generate checksums
  run: |
    cd artifacts
    sha256sum * > ../checksums.txt
```

This produces lines with bare filenames (no `artifacts/` prefix), which is simpler to parse in `install.sh`. **Prefer the `cd artifacts` form** so that `install.sh` can do a straightforward `grep "$BINARY" checksums.txt` without stripping a path prefix.

### Release asset upload

Both `artifacts/*` and `checksums.txt` are passed to `softprops/action-gh-release@v2` in the `files` field (handled by section-01-ci-workflow):

```yaml
files: |
  artifacts/*
  checksums.txt
```

`checksums.txt` is placed in the repository root (not inside `artifacts/`) so it is a distinct, clearly-named release asset alongside the binaries.

## Security Limitation

`checksums.txt` and the binaries are downloaded from the same GitHub Release. A compromised release would serve both a tampered binary and a matching (tampered) `checksums.txt`. This means the checksum:

- **Does protect against:** accidental download corruption, truncated transfers, CDN bit-rot
- **Does not protect against:** a compromised release (attacker controls both binary and `checksums.txt`)

GPG signing would address this gap by using a private key that is not stored in the repository. This is out of scope for this personal tool. The README (section-07-readme) and the install script output must disclose this limitation honestly so users who need stronger assurance know to build from source.

## Format Reference

The standard `sha256sum` output format is:

```
<64 lowercase hex characters><two spaces><filename>
```

Two spaces (not one) between the hash and the filename is the POSIX `sha256sum` convention. The `--check` flag expects this exact format. The Bats tests assert the two-space separator explicitly.

On macOS, `sha256sum` is not installed by default — `shasum -a 256` is the equivalent. The checksums generation step runs on `ubuntu-latest` in GitHub Actions, so `sha256sum` is always available there. The platform-specific tool selection concern applies only to `install.sh` verification (covered in section-04-install-core), not to the generation step.
